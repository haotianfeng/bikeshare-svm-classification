<a id="step-1-sampling"></a>
## 2. Step 1: 数据采样

### 模块：`src/sampling.py` + `src/data_loader.py`

### 原理

#### 问题

全年 660 万行数据对 SVM 来说太大——SVM 的训练时间复杂度为 $O(n^2)$ 到 $O(n^3)$。必须缩减样本量。

#### 方案：每月按比例分层抽样

1. **按比例分配**：先统计每月原始行数，按比例分配 60,000 个名额
   
   ```
   month_i_samples = total_samples × (month_i_rows / total_rows)
   ```
2. **每月内部分层**：对 member 和 casual 分别按比例抽样，保证每月内部类比例与原始一致
   
   ```python
   df.groupby("member_casual").apply(
       lambda g: g.sample(n=proportional_n, random_state=42)
   )
   ```

#### 为什么这样做

- **按比例**而非等量分配：保留数据的季节性——夏季 6 万行和冬季 3 万行的月度差异得以保留
- **分层**而非随机抽样：防止某些月份的散客样本过少（尤其冬季散客本就不多）
- **固定随机种子（42）** 保证每次运行结果一致，满足可复现性要求

#### 关键函数

| 函数                             | 位置                                          | 作用                                          |
| ------------------------------ | ------------------------------------------- | ------------------------------------------- |
| `discover_csv_files()`         | [data_loader.py:7](src/data_loader.py#L7)   | glob 发现 12 个月的 CSV 文件，返回 `{month: path}` 字典 |
| `load_single_month()`          | [data_loader.py:20](src/data_loader.py#L20) | 加载单月 CSV，解析时间戳，添加 month 列                   |
| `stratified_sample_by_month()` | [sampling.py:6](src/sampling.py#L6)         | 执行分层抽样，返回合并 DataFrame                       |
| `save_sampling_summary()`      | [sampling.py:58](src/sampling.py#L58)       | 保存每月抽样汇总表                                   |

#### 输出

- `outputs/tables/sampling_summary.csv` — 每月抽样行数与会员/散客分布

<a id="step-2-cleaning"></a>
## 3. Step 2: 数据清洗

### 模块：`src/cleaning.py`

### 原理

#### 3.1 DST（夏令时）修正

```
def fix_dst_negative_durations(df):
    mask = df["ended_at"] < df["started_at"]  # 结束 < 开始
    df.loc[mask, "ended_at"] += pd.Timedelta(hours=1)
```

**为什么需要**：2025 年 11 月美国从夏令时（EDT）切换回标准时间（EST），时钟回调 1 小时。如果骑行跨越了此时刻，可能出现数据库时间戳不一致导致`ended_at < started_at`。加回 1 小时即可修正。

#### 3.2 Haversine 球面距离

```python
def compute_haversine_km(lat1, lng1, lat2, lng2):
    a = sin²(Δlat/2) + cos(lat1)·cos(lat2)·sin²(Δlng/2)
    c = 2·arctan2(√a, √(1−a))
    return 6371 · c
```

**为什么不用欧氏距离**：华盛顿的纬度跨度约 0.1°（≈11km），在球面上直接用经纬度差近似会引入系统性偏差。Haversine 公式考虑了地球曲率，结果精确到米级。

#### 3.3 异常值处理策略

| 步骤             | 方法                    | 理由                            |
| -------------- | --------------------- | ----------------------------- |
| 时长 < 1 分钟      | 直接移除                  | 几乎不可能是真实骑行（可能是设备测试或误触）        |
| 时长 > 1440 分钟   | 直接移除                  | 24 小时以上极可能是忘记关锁               |
| 距离 = 0 但始终站点不同 | 直接移除                  | GPS 数据明显有误                    |
| 距离 IQR 异常值     | clip 到边界              | 保留样本量，用 winsorization 降低极端值影响 |
| 时长 IQR 异常值     | clip 到边界              | 同上                            |
| 平均速度异常         | clip 到 [0.5, 50] km/h | 0.5 km/h 以下≈步行，50 km/h 以上不合常理 |
| 终点经纬度缺失        | 移除行                   | 无法计算距离，且占比极小                  |

#### 为什么使用 IQR 缩尾而非直接删除

- 直接删除异常值会损失样本量（可能损失 5-15%）
- SVM 对极端值敏感，缩尾（winsorization）在不丢样本的前提下控制极端值影响
- 边界值（如 Q3 + 1.5×IQR）是统计学中广泛接受的异常值定义

#### 清洗顺序很重要

必须按以下顺序执行：

1. DST 修正 → 确保时长计算正确
2. 计算时长和距离 → 依赖修正后的时间戳
3. 移除极端值 → 避免对后续统计计算的影响
4. IQR 缩尾 → 此时 Q1/Q3 是基于已清理数据计算的

#### 输出

- `outputs/tables/cleaning_stats.csv` — 清洗前后对比

<a id="step-3-features"></a>
## 4. Step 3: 特征工程

### 模块：`src/feature_engineering.py`

### 原理

#### 4.1 循环编码（Cyclic Encoding）

**问题**：如果直接把"小时 23"和"小时 0"编码为 23 和 0，模型会认为它们的"距离"为 23——但实际只差 1 小时。

**方案**：将循环变量映射到单位圆的 (sin, cos) 坐标

```python
def _cyclic_encode(values, period):
    scaled = 2π · values / period
    return sin(scaled), cos(scaled)
```

**几何直觉**：

```
         12 (0,1)
    9 (-1,0)──┼──(1,0) 3
           (0,-1) 6
```

- 小时 0 (=24)→ (sin 0, cos 0) = (0, 1)
- 小时 6 → (sin π/2, cos π/2) = (1, 0)
- 小时 23 和小时 0 在圆上相邻，欧氏距离约为 2π/24 ≈ 0.26

#### 4.2 频率编码 vs One-Hot 编码

```
One-Hot: station_001=[1,0,0,...], station_002=[0,1,0,...], ...
        → 几百维的稀疏矩阵，维度灾难，SVM 训练极慢

频率编码: station_name → count(station) / total_rides
        → 1 维稠密特征，保留站点的"热门程度"信息
```

**权衡**：频率编码丢失了站点的身份信息——所有同频率的站点被视为相同。但对本任务的分类目标（区分用户类型而非预测具体站点），站点的热门程度比具体身份更重要。

#### 4.3 对数变换

```python
feats["duration_minutes_log"] = np.log1p(df["duration_minutes"])
```

**为什么**：骑行时长呈典型的长尾分布（大多数 < 20 分钟，少数 > 1 小时）。未变换的特征中，极少数长骑行会主导 SVM 的核函数计算。对数变换压缩长尾，使分布接近正态——这是 SVM（依赖欧氏距离）的理想特征形态。

#### 4.4 最终特征列表（17 维）

| 特征名                     | 类型  | 含义             |
| ----------------------- | --- | -------------- |
| `hour_sin`              | 循环  | 小时的 sin 分量     |
| `hour_cos`              | 循环  | 小时的 cos 分量     |
| `dayofweek_sin`         | 循环  | 星期几的 sin 分量    |
| `dayofweek_cos`         | 循环  | 星期几的 cos 分量    |
| `month_sin`             | 循环  | 月份的 sin 分量     |
| `month_cos`             | 循环  | 月份的 cos 分量     |
| `is_weekend`            | 二元  | 是否周末（周六/日=1）   |
| `is_morning_rush`       | 二元  | 工作日 7-9 时 =1   |
| `is_evening_rush`       | 二元  | 工作日 16-19 时 =1 |
| `duration_minutes_log`  | 连续  | 对数骑行时长         |
| `haversine_distance_km` | 连续  | 球面距离（km）       |
| `rideable_type`         | 二元  | 电动单车 =1        |
| `has_start_station`     | 二元  | 有无起始站点         |
| `has_end_station`       | 二元  | 有无终止站点         |
| `start_station_freq`    | 连续  | 起始站点频率         |

## 6. Step 5: 特征选择

### 模块：`src/feature_selection.py`

### 三种方法的原理

#### 6.1 互信息（Mutual Information）

$I(X; Y) = \sum_{x,y} P(x,y) \log \frac{P(x,y)}{P(x)P(y)}$

衡量特征 X 包含的关于目标 Y 的信息量（以 bit 为单位）。**优点**：捕获非线性关系，与模型无关。**缺点**：需要足够样本才能准确估计。

```python
mi_scores = mutual_info_classif(X, y, random_state=42)
```

#### 6.2 卡方检验（Chi-Square）

$\chi^2 = \sum \frac{(O_{ij} - E_{ij})^2}{E_{ij}}$

检验特征与类别之间的统计独立性。**前提**：特征必须非负（因此先用 MinMaxScaler 映射到 [0,1]）。**优点**：有明确的 p 值，统计意义清晰。

```python
X_01 = MinMaxScaler().fit_transform(X)
chi2_scores, p_values = chi2(X_01, y)
```

#### 6.3 随机森林 Gini 重要性

仅用于特征重要性分析（非主模型）。随机森林在训练过程中记录每个特征用于分裂时减少的 Gini 不纯度。**优点**：捕获非线性+交互效应。**仅用作评分的辅助工具**。

```python
rf = RandomForestClassifier(n_estimators=200, max_depth=10, random_state=42)
rf.fit(X, y)
# rf.feature_importances_ 即 Gini 重要性
```

#### 6.4 集成排名

三种方法给出三个排名 → 取平均排名作为最终共识排名：

```python
merged["mean_rank"] = (rank_mi + rank_chi2 + rank_rf) / 3
```

**为什么**：单一方法有其局限性——MI 对连续特征敏感但对二元特征不敏感，Chi2 要求非负值，RF 有随机性。平均排名减少单一方法偏差。

### 输出

- 4 张排名图（`feature_ranking_mi/chi2/rf/ensemble.png`）
- 4 张排名 CSV 表

<a id="step-6-7-split-scale"></a>
## 7. Step 6-7: 数据划分与标准化

### 模块：`src/preprocessing.py`

### 原理

#### 7.1 分层划分（Stratified Split）

```python
X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, stratify=y, random_state=42
)
```

**`stratify=y`** 是关键——确保训练集和测试集中会员/散客各占约 70%/30%。没有分层时，小样本情况下列分布可能出现随机波动，影响评估的公平性。

#### 7.2 Z-score 标准化

$x' = \frac{x - \mu}{\sigma}$

```python
scaler = StandardScaler()
X_train_scaled = scaler.fit_transform(X_train)  # 在训练集上拟合
X_test_scaled = scaler.transform(X_test)         # 仅变换测试集（无fit!）
```

**关键原则：防止数据泄露（Data Leakage）**

- `fit_transform` 只在训练集上调用——$\mu$ 和 $\sigma$ 仅从训练数据计算
- 测试集只用 `transform`——使用训练集的统计量来标准化
- 如果在划分前对整个数据集 `fit_transform`，则测试集的信息会"泄露"到训练过程中，导致过于乐观的评估结果

#### 7.3 为什么 SVM 必须要标准化

SVM 的 RBF 核函数基于欧氏距离：

$K(x_i, x_j) = \exp(-\gamma \|x_i - x_j\|^2)$

如果特征未经标准化：

- 数值范围大的特征（如距离，范围 0-10 km）会主导距离计算
- 数值范围小的特征（如频率，范围 0-0.01）几乎被忽略
- 模型实际上只用了"距离"一个特征，其他特征等于白构造了

标准化后每个特征对距离计算的贡献均等。

<a id="step-8-9-svm-modeling"></a>
## 8. Step 8-9: SVM 建模与超参数优化

### 模块：`src/train.py`

### 8.1 三种基准模型的设计意图

| 模型                                        | 核函数     | 设计意图                                 |
| ----------------------------------------- | ------- | ------------------------------------ |
| `LinearSVC(C=1.0)`                        | 线性      | 性能下界——最简单的线性边界，训练快（<1s），用于验证非线性核的必要性 |
| `SVC(kernel='poly', d=2)`                 | 多项式 d=2 | 中等非线性——检查低次多项式能否捕捉行为差异               |
| `SVC(kernel='rbf', C=1.0, gamma='scale')` | RBF     | 调优前的参考点——展示默认参数 RBF 的基线性能            |

三个模型均设置 `class_weight='balanced'`，自动按类别样本量的反比加权，缓解会员/散客 7:3 的不平衡。

### 8.2 GridSearchCV 的工作原理

```python
param_grid = {
    "C":     [0.01, 0.1, 1, 10, 100],    # 5 个值
    "gamma": ["scale", 0.1, 0.01, 0.001], # 4 个值
    "kernel": ["rbf"],                     # 1 个值
}
# 共 5×4×1 = 20 种组合
```

对于每一种参数组合：

1. 将训练集分为 5 折（5-fold CV）
2. 在 4 折上训练 SVM，在剩下 1 折上评估
3. 重复 5 次，每次换不同的验证折
4. 取 5 次准确率的平均值作为该组合的得分
5. 选平均得分最高的组合

```python
grid = GridSearchCV(svc, param_grid, cv=5, scoring="accuracy", n_jobs=-1)
grid.fit(X_train, y_train)
# grid.best_params_ → {"C": X, "gamma": X, "kernel": "rbf"}
# grid.best_score_  → 交叉验证最优得分
```

**设置 `n_jobs=-1`** 使 GridSearchCV 使用所有 CPU 核心并行搜索。

### 8.3 扩展搜索策略

如果最佳 C=100（搜索边界），则 C 可能更大更好：

```python
param_grid_extended = {
    "C": [100, 500, 1000],
    "gamma": ["scale", 0.1, 0.01],
}
```

这是一种务实的搜索策略——先粗搜索再精搜索，在有限的计算资源下覆盖更大的超参数空间。

### 8.4 最终模型

```python
final_model = SVC(
    kernel="rbf", C=best_C, gamma=best_gamma,
    class_weight="balanced", probability=True,  # probability=True 用于 ROC 曲线
    random_state=42, cache_size=2000,
)
```

`probability=True` 启用 Platt Scaling 进行概率校准（内部使用 5 折 CV 拟合 sigmoid 函数），使模型能输出 `predict_proba()` —— ROC 和 PR 曲线依赖此方法。

### 8.5 为什么用 LinearSVC 而非 SVC(kernel='linear')

`LinearSVC` 使用 liblinear 库，在大数据集上比 libsvm 的 `SVC(kernel='linear')` 快很多（本项目中 0.1s vs 可能几分钟）。`dual=False` + `max_iter=5000` 进一步优化了性能。

<a id="step-10-evaluation"></a>
## 9. Step 10: 模型评估

### 模块：`src/evaluate.py`

### 评估指标全面解析

#### 精确率与召回率的权衡

```
精确率 = TP / (TP + FP)  — 预测为"会员"的样本中，有多少真的是会员
召回率 = TP / (TP + FN)  — 真正的会员中，有多少被模型正确识别
F1    = 2PR / (P + R)    — 精确率与召回率的调和平均
```

对于本任务：精确率（会员）衡量"给散客误发会员优惠券"的风险，召回率（会员）衡量"遗漏潜在会员"的代价。

#### ROC-AUC 与 PR-AUC

- **ROC-AUC**：在所有可能的分类阈值下，TPR vs FPR 曲线下的面积。反映模型整体排序能力。类别不均衡时仍然乐观。
- **PR-AUC**：Precision vs Recall 曲线下面积。类别不均衡时比 ROC 更反映真实性能——因为 Precision 受类别比例影响。

#### 学习曲线

```python
train_sizes, train_scores, val_scores = learning_curve(
    model, X, y, cv=5, train_sizes=np.linspace(0.1, 1.0, 10)
)
```

学习曲线展示训练集和验证集准确率随样本量增加的变化：

- **两条线收敛且高** → 模型拟合良好
- **训练集远高于验证集（大 gap）** → 过拟合
- **两条线都低** → 欠拟合

## 10. Step 11: 特征重要性分析

### 模块：`src/feature_analysis.py`

### 10.1 排列重要性（Permutation Importance）

```python
result = permutation_importance(model, X_test, y_test,
    n_repeats=30, scoring="accuracy", random_state=42)
```

**原理**：

1. 用训练好的模型在测试集上预测，记录基准准确率
2. 将第 j 个特征的值**随机打乱**，打破该特征与标签的关联
3. 重新预测，准确率下降的幅度 = 该特征的重要性
4. 重复 30 次（`n_repeats=30`），取均值和标准差

**为什么 n_repeats=30**：单次打乱有随机性，多次重复取均值更稳定，标准差衡量重要性的稳定性。

**为什么不在训练集上做**：训练集上模型已"记住"了模式，打乱特征的影响可能被低估。

### 10.2 逻辑回归系数（辅助验证）

```python
lr = LogisticRegression(penalty="l2", C=1.0, class_weight="balanced", max_iter=5000)
lr.fit(X_train, y_train)
```

在标准化特征上，逻辑回归系数的大小反映特征对 log-odds 的线性影响，符号反映方向（正→会员倾向，负→散客倾向）。虽然逻辑回归只能捕捉线性关系，但其系数提供了一种直观的解释。

## 11. Step 12: 有趣发现与统计检验

### 6 项统计检验

| 检验   | 方法             | 零假设 H₀             |
| ---- | -------------- | ------------------ |
| 通勤模式 | 卡方检验           | 会员与散客在高峰/非高峰的分布无差异 |
| 时长差异 | Mann-Whitney U | 会员与散客的骑行时长分布相同     |
| 周末效应 | 卡方检验           | 工作日与周末的散客比例无差异     |

| 电动偏好 | 卡方检验           | 用户类别与单车类型偏好独立      |

### 为什么用 Mann-Whitney U 而非 t 检验

骑行时长呈长尾分布，不满足 t 检验的正态分布假设。Mann-Whitney U 是非参数检验，仅要求独立样本和顺序尺度，对分布形状无要求。

## 13. 关键技术决策总结

| 决策点   | 选择                      | 理由                    |
| ----- | ----------------------- | --------------------- |
| 抽样大小  | 60,000                  | 权衡 SVM 训练时间与样本代表性     |
| 时间编码  | sin/cos 循环编码            | 保留循环拓扑，比直接数值编码好       |
| 站点编码  | 频率编码                    | 避免高维 One-Hot，保留热门程度信息 |
| 时长变换  | log1p                   | 压缩长尾，帮助 SVM 的欧氏距离度量   |
| 异常值处理 | IQR 缩尾                  | 保留样本量，比直接删除温和         |
| 核函数   | RBF                     | 最佳非线性表达能力，单参数易调优      |
| 参数搜索  | GridSearchCV + 扩展搜索     | 先粗后精，覆盖大范围            |
| 类别不平衡 | class_weight='balanced' | 比 SMOTE 等方法简单有效       |
| 特征重要性 | 排列重要性                   | 与模型无关，直接度量预测影响        |
| 统计检验  | 卡方 + Mann-Whitney U     | 适配分类和连续数据，不依赖分布假设     |

> 本章节记录从原始代码到当前版本的每一次重要修改、发现的 Bug 及根因分析、优化决策及最终结论。四个轮次的迭代顺序阅读效果最佳。

### 14.1 第一轮：32GB 内存爆炸修复

#### 问题现象

在 i9-13900HX (32 线程) + 32GB RAM + RTX 4080 Laptop 上初次运行 `main.py`，约 5 分钟后内存占用飙升至 32GB+，Windows 开始 swap，磁盘 I/O 100%，系统完全卡死，需强制关机。

#### 根因：四个内存杀手叠加

**杀手 1（最致命）：`n_jobs=-1` 进程 fork 炸弹**

```python
# train.py 原代码
GridSearchCV(svc, param_grid, cv=5, n_jobs=-1)
```

i9-13900HX 有 32 个逻辑核心，`n_jobs=-1` 使 sklearn 的 joblib fork 出 32 个子进程。每个子进程是 fork 出来的，复制父进程的整个内存空间。Python 的引用计数机制导致 copy-on-write 几乎立即触发实际内存复制，每个子进程实际消耗接近完整副本的内存（~1-2 GB），32 个进程合计 > 32 GB。

GridSearchCV 的默认 `pre_dispatch='2*n_jobs'` = 64 个任务预分配，进一步加剧了问题。

**杀手 2：`cache_size=2000` 的 SVC 内核缓存**

```python
# 原代码中 6 处 SVC 实例化均硬编码 cache_size=2000
SVC(cache_size=2000)  # 2000 MB = 2 GB per instance
```

GridSearchCV 期间最多 32 个 SVC 实例同时拟合，每个请求 2GB 内核缓存：

```
峰值缓存占用: 32 × 2 GB = 64 GB → 仅缓存就超出物理内存
```

加上 SVC 拟合时实际的核矩阵（O(n²) 规模），内存需求远超 32GB。

**杀手 3：全量 CSV 加载**

```python
# sampling.py 原代码
for month, path in files.items():
    df = load_single_month(path)  # 加载整月 70 万行
    # 然后从中只取 ~5000 行...
```

单月 70 万行 × 13 列（含长字符串站点名）的 DataFrame 约 1.2 GB。Python GC 不及时回收，可能和下个月的加载叠加。

**杀手 4：其他并行热点**

- `learning_curve(n_jobs=-1)` — 又 32 进程
- `RandomForestClassifier(n_estimators=200, n_jobs=-1)` — 200 棵树 × 32 进程
- `permutation_importance(n_repeats=30, n_jobs=-1)` — 30 轮 × 32 进程

#### 完整修改清单

| 文件 | 修改项 | 旧值 | 新值 |
|------|--------|------|------|
| `config.py` | 新增 `N_JOBS` | 无 | `4` |
| `config.py` | 新增 `SVC_CACHE_SIZE` | 无 | `500` |
| `config.py` | `TOTAL_SAMPLE_SIZE` | `60000` | `40000` |
| `train.py` | GridSearchCV `n_jobs` | `-1` | `N_JOBS` (4) |
| `train.py` | GridSearchCV `pre_dispatch` | 默认(64) | `"2*n_jobs"` (8) |
| `train.py` | 6 处 `SVC(cache_size=...)` | `2000` | `SVC_CACHE_SIZE` (500) |
| `evaluate.py` | `learning_curve(n_jobs=...)` | `-1` | `N_JOBS` (4) |
| `feature_selection.py` | `RandomForest(n_estimators=...)` | `200` | `100` |
| `feature_selection.py` | `RandomForest(n_jobs=...)` | `-1` | `N_JOBS` (4) |
| `feature_analysis.py` | `permutation_importance(n_repeats=...)` | `30` | `15` |
| `feature_analysis.py` | `permutation_importance(n_jobs=...)` | `-1` | `N_JOBS` (4) |
| `sampling.py` | CSV 加载方式 | 全量加载 | `chunksize=100000` 分块 |
| `sampling.py` | 列过滤 | 全部列 | `usecols=13` 列 |
| `main.py` | 内存监控 | 无 | `_mem_mb()` + `gc.collect()` |

#### 修改后峰值估算

```
GridSearchCV 阶段:
  4 进程 × (500MB 缓存 + ~200MB 数据) = ~2.8 GB
  Python + DataFrame                   = ~1.5 GB
  总峰值                                ≈ 4-5 GB  ✅
```

### 14.2 第二轮：GridSearchCV 耗时过长

#### 问题现象

第一轮修复后内存不再爆炸，但 Step 9 (GridSearchCV) 运行超过 1 小时仍未完成，CPU 占用 100%。

#### 根因

第一轮将样本量从 6 万降到 4 万，**内存安全了但计算量仍然巨大**。

RBF 核 SVM 的训练复杂度为 $O(n^2) \sim O(n^3)$：

```
训练样本: 40,000 × 80% = 32,000 条
参数网格: 5 C × 4 gamma = 20 组合
5-fold CV: 20 × 5 = 100 次 SVC.fit()

单次 SVC.fit() 耗时 (32,000 样本):
  C=0.01  → ~30-60 秒
  C=100   → ~120-300 秒（大 C → 更多支持向量 → 收敛极慢）

100 次拟合总耗时: 可能 > 1 小时
```

**关键认知**：SVM 不是神经网络 — 不需要海量数据。数据量超过阈值后边际收益极低，但计算成本指数增长。

#### 修改

`TOTAL_SAMPLE_SIZE`: `40000` → `12000`

#### 结果

- 总耗时: 3.6 分钟 (原 > 1 小时)
- 内存峰值: 350 MB
- Best params: `C=0.01, gamma=0.1`
- **发现：C=0.01 在搜索边界上** — 真正的最优值可能在更小或更大的方向

### 14.3 第三轮：5 个 Bug 修复

#### Bug 1：Step 12 崩溃 — 特征列名冲突（最严重）

**现象**：

```
ValueError: No data; `observed` has size 0.
  at findings.py:146 → analyze_electric_preference()

**根因**：`feature_engineering.py` 创建了名为 `rideable_type` 的特征列（值 0/1），与原始 DataFrame 中同名字符串列冲突。`main.py` 将特征写回 `df_clean` 时覆盖了原列：

feats["rideable_type"] = (df["rideable_type"] == "electric_bike").astype(int)

# main.py — 覆盖发生处
for i, name in enumerate(feature_names):
    df_clean[name] = X[:, i]  # df_clean["rideable_type"] 从字符串变成 0/1！
```

之后 `findings.py` 尝试匹配 `"electric_bike"` 时全部命中 NaN，crosstab 为空，`chi2_contingency()` 抛异常。

**修复**：将特征名从 `rideable_type` 改为 `is_electric_bike`。

**教训**：特征命名时避免与原始数据列同名，尤其是特征会被写回原 DataFrame 的场景。

#### Bug 2：程序退出时 tkinter 线程报错

**现象**：`RuntimeError: main thread is not in main loop` × 5 次

**根因**：Windows 下 matplotlib 默认 tkinter 后端，程序退出时在主线程外清理 GUI 资源。

**现象**：`FutureWarning: The probability parameter was deprecated in 1.9`

**修复**：`SVC(probability=True)` → `CalibratedClassifierCV(SVC(), cv=5, ensemble=False)`。内部使用 Platt scaling（5 折 CV 拟合 sigmoid）进行概率校准。

#### Bug 5：sklearn LogisticRegression penalty 废弃警告

**修复**：移除 `penalty="l2"`（已是新版默认值）。

### 14.4 第四轮：模型精度优化

#### 问题分析

第三轮后程序稳定运行，但模型质量不理想：

| 指标 | 值 | 问题 |
|------|-----|------|
| Test Accuracy | 71.33% | 仅比 70% 多数类基线高 1.3% |
| Casual Recall | 27.31% | 模型几乎不识别散客 |
| Member Recall | 92.03% | 模型几乎全猜 member |
| Best C | 0.01 | **在搜索边界上** |
| GridSearchCV scoring | `accuracy` | 不均衡数据下偏向多数类 |

模型在"偷懒" — 它发现把所有样本预测为 member（约占 70%）就能获得 ~70% accuracy，GridSearchCV 用 accuracy 评分反而奖励了这种行为。

#### 优化方案

| 修改项 | 旧值 | 新值 | 理由 |
|------|------|------|------|
| `TOTAL_SAMPLE_SIZE` | `12000` | `20000` | 更多散客样本（~5500），给少数类更多学习机会 |
| `PARAM_GRID["C"]` | `[0.01, 0.1, 1, 10, 100]` | `[0.001, 0.005, 0.01, 0.05, 0.1, 1, 10, 100]` | C=0.01 在边界，探索更小值 |
| `PARAM_GRID["gamma"]` | `['scale', 0.1, 0.01, 0.001]` | `['scale', 0.1, 0.05, 0.01, 0.001]` | 增加 0.05 粒度 |
| `SCORING`（新增） | 硬编码 `"accuracy"` | `"f1_macro"` | 平等对待两个类别，不再奖励偏斜 |
| PARAM_GRID_EXTENDED | C=`[100,500,1000]` | C=`[100,200,500,1000]` | 增加 200 粒度 |
| RF `n_estimators` | `100` | `150` | 样本增加，RF 更稳定 |
| Permutation `n_repeats` | `15` | `20` | 更大样本下更多重复 |

#### 关于 scoring='f1_macro' 为什么关键

在 70/30 不均衡数据下：

```
scoring='accuracy':
  模型 A: 全猜 member → accuracy ≈ 70% → 高分
  模型 B: 努力区分，但 Casual Recall 低 → accuracy ≈ 68% → 低分
  → GridSearchCV 选模型 A，实际上是个废模型

scoring='f1_macro':
  模型 A: F1_macro = (F1_member=~0.82 + F1_casual=0) / 2 ≈ 0.41
  模型 B: F1_macro = (F1_member=~0.80 + F1_casual=~0.35) / 2 ≈ 0.58
  → GridSearchCV 选模型 B，真正会区分的模型
```

$F1_{macro} = \frac{1}{K}\sum_{k=1}^{K} F1_k$，其中 $K=2$（member, casual）。简单平均意味着每个类别权重相等，与类别样本量无关。

#### 最终结果

```
总耗时:         15.3 分钟（可接受）
内存峰值:        357 MB（远低于 32GB）
GridSearchCV:   200 fits, 10.4 分钟

Best params:    C=0.05, gamma=0.1  ← C 不再卡边界！
Test Accuracy:  73.27%
Casual Recall:  27.46%   ← 仍然很低
Member Recall:  92.28%
ROC-AUC:        0.6928
```

#### 核心发现：特征区分度存在上限

**即使翻倍样本量、扩大搜索范围、换成 f1_macro 评分，散客召回率仍然只有 27%。这不是参数问题，是特征区分度不够。**

原因分析：

1. **两类用户都在骑共享单车** — 时间、距离、速度的分布有大量重叠
2. **Member 有规律但 Casual 也有规律** — 游客在景点附近、休闲骑行者在周末、偶尔通勤者……casual 并非无规律，而是规律更分散
3. **17 个特征主要描述"如何骑行"（how）** — 而非"为什么骑行"（why）。缺失关键上下文：是否有季票、骑行目的、用户注册时长等

这是一个**有价值的实验发现**，在报告中可以这样定位：

- 73.27% > 70% 基线 → 模型确实学到了区分模式
- 散客识别难 → casual 行为高度多样化，不是简单的"非通勤者"
- ROC-AUC 0.69 → 有一定排序能力，但区分度有限
- 特征重要性稳定一致 → `duration_minutes_log`、`avg_speed_kmh` 始终是 Top 特征
- 这也解释了为什么仅凭骑行行为数据难以精确预测用户类型
